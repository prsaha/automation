# OLP Overage Revenue Query — TDD Gap Analysis
**Query:** `olp_overage_revenue.sql`
**TDD:** RD-1160536 — Pricing for Dbt Run Cache
**Reviewed:** 2026-03-09

---

## Gap 1: `stripe_customer_id` will break after migration

**Severity:** High
**Line:** 5, 15

The query reads `stripe_customer_id` from `accounts`:
```sql
accounts.stripe_customer_id
...
inner join `digital-arbor-400`.`pg_public`.`accounts` on revenue_records.billing_account_id = accounts.id
```

The TDD migration drops this column from `accounts` and moves it to `account_billing_info`:
```sql
-- TDD migration step
DROP COLUMN IF EXISTS stripe_customer_id;  -- on accounts table
```

**Required fix:** Replace the `accounts` join (or add a second join) to `account_billing_info` and read `stripe_customer_id` from there:
```sql
left join `digital-arbor-400`.`pg_public`.`account_billing_info`
  on account_billing_info.account_id = revenue_records.billing_account_id
  and account_billing_info.billing_account_type = 'FIVETRAN'
```

## Gap 2: RunCache records excluded implicitly, not explicitly

**Severity:** Medium
**Line:** 15

The `INNER JOIN accounts` silently drops RunCache records because their `billing_account_id` maps to `account_billing_info.id` (with `external_` prefix), not `accounts.id`. This happens to produce the correct scoping for now, but:

- It relies on an implicit side effect of the join rather than intentional filtering
- If a RunCache account ever gets a matching `accounts` row, records would leak into this OLP pipeline

**Required fix:** Add an explicit `product_type` guard to the WHERE clause once the column is available:
```sql
and (revenue_records.product_type is null or revenue_records.product_type != 'DBT_RUN_CACHE')
```

## Gap 3: NS customer join key is suspect

**Severity:** High
**Line:** 44

The query joins `netsuite2.customer` on:
```sql
olp_overage_revenue.billing_account_id = customer.custentity_ft_account_id_sf
```

The field name `custentity_ft_account_id_sf` has an `_sf` suffix, strongly suggesting it stores a **Salesforce account ID**, not a `billing_account_id`. If that is the case:

- The join condition is incorrect — `billing_account_id` (a Fivetran internal ID) will not match a SFDC-derived field
- All NS customer lookups would return NULL, producing invoices with no customer, no `external_id`, and no `to_be_emailed`

**Required action:** Confirm what `custentity_ft_account_id_sf` actually stores in the `netsuite2` connector. If it is a SFDC ID, the join key needs to be corrected to the appropriate Fivetran account ID field in NetSuite.

## Gap 4: No `product_type` separation for Finance reporting

**Severity:** Medium
**Line:** n/a

The TDD success criterion states:
> "The Finance team can separate revenue generated by Fivetran from revenue generated by RunCache"

The query has no `product_type` column in the SELECT and no filter by product type. Even though RunCache records are currently excluded by the INNER JOIN, the output contains no field that allows Finance to distinguish OLP revenue from other product lines in downstream reporting.

**Required fix:** Add `revenue_records.product_type` to the SELECT in CTE 1 and propagate it through to the final output.

## Gap 5: `custentitycustentity_email` — likely column name typo

**Severity:** Medium
**Lines:** 35, 38, 40

The `to_be_emailed` logic references `customer.custentitycustentity_email` — `custentity` appears doubled:
```sql
if(
  customer.email = customer.custentitycustentity_email,
  customer.email,
  coalesce(
    concat(customer.email, ',', customer.custentitycustentity_email),
    ...
  )
)
```

If this column does not exist, the query will fail at runtime. If it silently returns NULL (e.g. BigQuery permissive mode), all `to_be_emailed` values would fall back to `customer.email` only, silently dropping additional email recipients.

**Required fix:** Confirm the correct column name in `netsuite2.customer` (likely `customer.custentity_email`).

## Gap 6: `BigQueryToAppSyncNetsuiteFinancials` — external ID format

**Severity:** Low
**Lines:** 96–109

The TDD requires `BigQueryToAppSyncNetsuiteFinancials` to sync invoices via `account_billing_info` rather than `accounts`. The `external_id` is constructed as:
```sql
concat('CBPINV', netsuite_customer_id, cast(revenue_month_end as string format 'YYYYMM'))
```

For RunCache accounts (once supported), `netsuite_customer_id` may be NULL (no NS customer lookup path exists for external accounts in this query), producing a NULL `external_id`. The downstream NS write needs to handle the `external_` prefix ID format from `account_billing_info`.
